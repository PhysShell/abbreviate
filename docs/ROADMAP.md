# Дорожная карта и приоритеты

Статус ядра: пять слоёв работают на реальных данных (48k форм, леммы,
биграм-LM +13пп, SymSpell-опечатки, слой сокращений/сленга, метрики, web-демо).
`tune` подтвердил, что веса near-optimal. Узкое место теперь — **не алгоритмы**,
а попадание в реальный поток набора (Android) и реальные данные (логи принятий).

Сквозной принцип приоритизации:

> Бо́льшая часть «умных» фич — это один **контур обратной связи** в разных масках
> (undo, обучение на откатах, lifecycle принятия, harm-rate, исключения). Тот же
> урок, что и в бенчмарках: **«тапнул» ≠ «доволен»**. Этот сигнал нужно заложить
> правильно и рано — он необратим: ранние логи с неверной семантикой бесполезны.
> Всё остальное либо дешёвые источники-кандидаты (данные на готовом слое), либо
> другой продукт (грамматика), либо зашлюзовано на реальные данные (ML).

## 🟢 Заложить сейчас (дёшево, фундаментально)

1. **Жизненный цикл коррекции в сигнале принятия.** Сейчас `UserHistory` —
   плоский счётчик `(скелет→форма)→count` с изъяном «тап = успех». Нужно:
   негативный сигнал (откат → понижение приора) и различие
   `confirmed / reverted / edited`; в ранжировании учитывать **confirmed**, не
   tapped. Расширить событие opt-in лога web-демо до
   `{shown, chosen, rank, status}`. Это единственное, ради чего трогаем ядро
   прямо сейчас. — *частично в работе; см. ниже.*
2. **Спека приватного логгинга для Android** (хеши/длины/метрики, сырой текст —
   только по явному opt-in; crash-bundle с preview). Зафиксировать принцип, не
   реализовывать.

## 🟡 Взять вместе с Android-оболочкой (продуктовый шаг)

3. **Android IME** — главный шаг к догфудингу. Загрузка артефактов из assets,
   лента подсказок, hold-формы, персистентная история, настройки. Код пишется,
   но сборка/проверка APK — вне этой среды.
4. **Undo / «показать оригинал» + обучение на откатах + исключения
   («не исправлять это»).** Всё завязано на **автозамену**, которой ещё нет
   (сейчас только tap-to-commit). Автозамена — по margin-политике
   (`top1−top2 > порог` + длина + не-protected + история), suggest-first.
5. **Транслит-термины `гитхаб → GitHub`** — это **тот же слой сокращений**, не
   новая подсистема. Данные: `data/translit/ru-tech.tsv` (готово). Хорошая
   догфуд-фича для dev-аудитории. Контекст-буст («GitHub выше рядом с „пуш“») —
   переиспользование LM, отложить.

## 🟠 Отложить / отдельный трек (не уводить core)

6. **Морфология (граммемы OpenCorpora).** Сигнал `morph_compatibility`,
   группировка hold по падежам, теги форм. Крупнейший алгоритмический прирост для
   русского, но мерить эффект придётся синтетикой, пока нет п.1-данных.
7. **Грамматика `в это случае → в этом случае`** — **другой продукт**
   (LanguageTool/Grammarly): согласование на спане, нужна морфология, худший
   режим отказа — молча портить правильный текст. Решение:
   - фиксированные фразы (`по крайне мере → по крайней мере`) = тривиальные
     данные на слое сокращений, можно когда угодно;
   - общее падежное согласование = далёкое будущее, после морфологии, **только
     подсказкой**, со своим eval и precision-first. Возможно, никогда.
8. **Короткий режим (2 буквы, `тя→тебя`)** — данные на слое сокращений
   (`ru-short.tsv`), не подсистема. Ценность узкая (служебные слова), валидация
   догфудингом, не бенчмарком. Делать вместе с Android.

## 🔴 Зашлюзовано на реальные данные (после догфудинга)

9. **Обучаемый реранкер** (логрегрессия/GBM на фичах из реальных принятий).
   `tune` показал: на синтетике переобучается → имеет смысл **только** на живых
   логах. Иначе просто переучивает генератор.
10. **SLM / GECToR-теггер / on-device нейросеть.** Никакой модели в hot-path;
    LLM только офлайн-учитель. Зашлюзовано на п.9, который на п.3.

## ✅ Что LLM-учитель даёт уже сейчас (офлайн, дёшево)

- **Транслит-алиасы** — `data/translit/ru-tech.tsv` (готово; промпт для
  расширения — в шапке файла).
- **Hard-negatives для бенчмарка** — `data/bench/adversarial.tsv`: реальные
  агрессивные сокращения (из майнинга чат-корпуса) + «не трогать» кейсы, которых
  синтетический генератор не делает. Делает бенчмарк честнее.

## Майнинг реальных данных (когда будет чат-корпус)

Доказано на субтитрах: из неформального текста можно автоматически добывать пары
`сокращение → слово` (`зн→знаю`, `гд→где`, `чув→чувак`). Шум — имена/иностранное;
нужен фильтр (имена с большой буквы, частотный порог, разные контексты). Дамп
Telegram-канала / `IlyaGusev/pikabu` (HF) / Taiga — прогоняются тем же майнером.
Ценность: калибровка генератора, доменные частоты, реальный eval — **не** «новые
очевидные сокращения» (частотные уже в лексиконе).
